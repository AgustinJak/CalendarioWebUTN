Procesos

La abstracción central de cualquier sistema operativo. Todo lo demás depende de este concepto.

Glosario de arranque
Términos esenciales

Si venís de otro rubro, estos términos aparecen constantemente. Leelos una vez antes de seguir.

Proceso
Un programa que está corriendo ahora mismo. No el archivo del programa guardado en disco, sino la "instancia viva" con memoria asignada y estado propio.
ej: tenés abierto Chrome → eso es (al menos) un proceso activo
Programa
El archivo estático guardado en disco con las instrucciones. Es pasivo. Un mismo programa puede generar múltiples procesos simultáneos.
ej: chrome.exe en tu disco duro = programa. Cada pestaña = proceso separado
Dirección de memoria
Un número que identifica una celda específica dentro de la memoria RAM, como un número de departamento en un edificio. Se escribe en hexadecimal por convención.
0x00401A3F → casillero nro. 4,201,023 de la RAM (en decimal)
Hexadecimal (0x…)
Sistema de numeración en base 16 que usan las computadoras por ser más compacto que el decimal. El prefijo "0x" avisa que lo que sigue es hexadecimal.
0xFF = 255 en decimal · 0x10 = 16 en decimal
Registro del procesador
Espacio de almacenamiento ultra-rápido dentro del chip del procesador. Guarda valores temporales durante la ejecución. Mucho más rápido que la RAM pero hay muy pocos.
R0, R1… R7 = registros de propósito general · PC = registro especial con la dirección de la próxima instrucción
PC (Program Counter)
Registro especial que contiene la dirección de memoria de la próxima instrucción que el procesador va a ejecutar. Avanza automáticamente después de cada instrucción.
Si PC = 0x004A → el procesador irá a buscar la instrucción guardada en el casillero 0x004A de la RAM
PSW (Program Status Word)
Un registro especial con bits que describen el estado actual del procesador: si hubo overflow, si el resultado fue cero, el modo de ejecución (usuario/kernel), etc.
PSW = 0b00100110 → cada bit tiene un significado específico (como flags de semáforo)
Syscall (llamada al sistema)
Cuando un programa necesita algo del SO (leer un archivo, crear otro proceso, usar la red), ejecuta una "syscall": le pide al SO que haga esa tarea por él.
open("archivo.txt") en tu código → internamente es una syscall al SO
PID (Process ID)
Número único que el SO le asigna a cada proceso cuando lo crea. Sirve para identificarlo y referenciarlo. Como un DNI del proceso.
En Linux/Mac: escribí "ps aux" en terminal para ver los PIDs de tus procesos activos
BCP / PCB
Bloque de Control de Proceso (Process Control Block en inglés). Estructura de datos que el SO crea y mantiene con toda la info de un proceso: su estado, registros, archivos abiertos, etc.
Es la "ficha técnica" del proceso. Sin ella, el SO no sabría cómo retomar su ejecución tras una pausa.
Stack (Pila)
Zona de memoria que crece y decrece automáticamente. Guarda el contexto de cada función llamada: sus variables locales, parámetros y la dirección de retorno.
Llamás a función A(), que llama a B() → el stack tiene A encima de B. Cuando B() termina, se "desapila" y volvés a A.
Heap (Montículo)
Zona de memoria que el programa gestiona manualmente para estructuras de tamaño variable que viven más que una función. En C se usa malloc(), en Java/Python es automático.
Creás una lista que puede crecer → esa memoria viene del heap · Se libera con free() en C o automáticamente en lenguajes modernos
01 · Definición
Imagen del proceso en memoria

Cuando el SO carga un proceso, lo organiza en zonas bien definidas dentro de la RAM. Cada zona tiene un propósito específico.

↑ DIRECCIÓN ALTA
Contexto
PC, PSW, registros
Stack
pila de llamadas
↓ crece hacia abajo
⋯ espacio libre ⋯
Heap
memoria dinámica
↑ crece hacia arriba
Datos
variables globales
Texto
instrucciones (código)
↓ DIRECCIÓN BAJA
Contexto Fotografía del estado del procesador en el momento exacto en que el proceso fue pausado. Contiene PC (¿en qué instrucción íbamos?), PSW (¿qué flags había?) y el valor de todos los registros.
Stack (Pila) Crece hacia abajo. Cada vez que tu código llama a una función, se apila un "frame" con sus variables locales y la dirección de retorno. Al salir de la función, se desapila.
Heap Crece hacia arriba. Es la memoria que pedís explícitamente cuando necesitás estructuras de tamaño variable en tiempo de ejecución (listas, árboles, objetos).
Datos Variables globales y estáticas. Su tamaño se conoce al compilar y no cambia durante la ejecución.
Texto Las instrucciones del programa en código máquina, tal como las leyó el SO desde el archivo en disco. Solo lectura: el programa no puede modificar su propio código.
⚠ Programa ≠ Proceso. El programa es el archivo .exe (o binario) guardado en disco. El proceso es ese programa corriendo en memoria, con sus propios datos y contexto. Un mismo programa puede originar múltiples procesos simultáneos (ej: varias ventanas del mismo editor).

03 · Ciclo de vida
Estados de un proceso

El SO modela la vida de cada proceso como una máquina de estados. Los modelos evolucionaron para resolver problemas concretos. Explorá cada uno:

Problema que tiene
Una cola para todo
Los procesos listos para correr y los que están bloqueados esperando el disco comparten la misma cola. El dispatcher no puede distinguirlos y pierde tiempo revisando procesos que no pueden ejecutar.
Dispatcher
¿Qué es el dispatcher?
Un pequeño programa dentro del SO encargado de hacer el intercambio: pausa el proceso actual, guarda su BCP y activa el siguiente de la cola. Es el "semáforo" del procesador.
Solución
→ Pasar al modelo de 5
Dividir "No Ejecutando" en dos estados ("Listo" y "Bloqueado") permite al SO saber exactamente cuáles pueden recibir CPU ahora y cuáles no.
Nuevo → Listo
Admisión
El SO creó el BCP y asignó memoria. El proceso ya está en RAM listo para correr. El SO puede demorar este paso si la RAM está muy cargada.
Ejecutando → Bloqueado
Espera de E/S
El proceso pidió leer un archivo. Leer del disco tarda ~1 millón de veces más que una instrucción de CPU. En lugar de esperar, el SO pausa el proceso y activa otro.
Ejecutando → Listo
Temporización
El timer expiró. El proceso agotó su quantum (ración de tiempo). Vuelve a la cola de Listos para dar paso al siguiente. Puede ser reactivado pronto.
Problema que resuelve
RAM llena, todos bloqueados
Sin swap: si todos los procesos en RAM están esperando E/S, el procesador queda ocioso y no se pueden admitir procesos nuevos. El swap libera RAM moviéndolos al disco.
Listo/Suspendido
Listo pero en disco
El proceso no está esperando ningún evento — podría correr ahora mismo — pero su imagen fue movida al disco para liberar RAM. Al reactivarse, se carga de vuelta y pasa a Listo.
Concepto clave
Swap es una operación de E/S
Mover un proceso entre RAM y disco es costoso (puede tardar varios milisegundos). El SO lo hace solo cuando no queda otra opción y usa estadísticas para minimizarlo.
Un proceso pausado debe poder reanudarse exactamente donde fue interrumpido, como si nunca hubiera parado. — Principio del context switch
04 · Mecanismo central
Context switch — paso a paso

Un cambio de contexto ocurre cada vez que el SO recupera el control del procesador. Puede ser por interrupción de timer, por E/S completada, o por una syscall del propio proceso.

1
El procesador detecta la señal de interrupción
Al terminar la instrucción en curso, el procesador revisa su registro de interrupciones. Si hay una señal pendiente (timer expirado, disco listo, syscall), suspende el ciclo normal de fetch-decode-execute.
2
Cambio de modo: usuario → kernel (modo núcleo)
Un bit dentro del PSW cambia de 0 (modo usuario, restringido) a 1 (modo kernel, privilegiado). Ahora el procesador puede ejecutar instrucciones del SO que el proceso de usuario no podía.
-- Antes de la interrupción:
PSW = 0b00100110 ← bit de modo = 0 (usuario)

-- Después, al entrar al SO:
PSW = 0b00101110 ← bit de modo = 1 (kernel)
3
El SO guarda el contexto completo del proceso actual en su BCP
Todos los registros del procesador se copian a la entrada del proceso en la process table. Es como sacar una fotografía exacta del estado de la CPU en este instante.
-- SO guarda en BCP[PID=4821]:
PC ← 0x00401A3F ← "íbamos en el casillero 4,201,023 de RAM"
PSW ← 0b00100110 ← estado de los flags
R0 ← 0x07, R1 ← 0x0D ... ← valores de los registros
4
El SO atiende la interrupción y el scheduler elige el próximo proceso
Si fue una interrupción de E/S, el SO mueve los procesos que esperaban ese evento a estado Listo. Luego el scheduler revisa la cola de Listos y elige el de mayor prioridad.
5
Se restaura el contexto del proceso elegido desde su BCP
Los valores guardados en el BCP del proceso elegido se cargan en los registros del hardware. El PC apunta exactamente donde ese proceso había quedado la última vez.
-- SO carga desde BCP[PID=3210] al hardware:
Procesador.PC ← 0x00802B10 ← continuará desde acá
Procesador.PSW ← 0b01001100 ← su estado de flags
Procesador.R0 ← 0x2A, R1 ← 0x00 ...
6
Modo kernel → usuario. El proceso continúa como si nada
El bit de modo en PSW vuelve a 0. El nuevo proceso ejecuta la instrucción a la que apunta su PC. Para él, el tiempo simplemente "pasó" — no sabe que fue suspendido.
05 · Hardware
Modos de ejecución

El procesador tiene dos "niveles de confianza". Esta separación es lo que impide que un proceso de usuario rompa el sistema operativo.

Modo Usuario — bit PSW = 0

Ejecutan todos los programas comunes. Acceso restringido: no pueden ejecutar instrucciones que afecten a otros procesos ni acceder directamente al hardware.

Si un proceso intenta ejecutar una instrucción privilegiada en modo usuario → excepción (trap). El SO toma el control e investiga qué pasó.

Bit de modo:
0
= modo usuario (restringido)
Modo Kernel / Núcleo — bit PSW = 1

Solo ejecuta el sistema operativo. Control absoluto sobre el procesador: puede ejecutar cualquier instrucción, acceder a cualquier dirección de memoria, modificar el hardware.

Se activa automáticamente al recibir una interrupción, excepción o syscall. Al terminar la rutina del SO, vuelve a modo usuario.

Bit de modo:
1
= modo kernel (privilegiado)
06 · Manejo de errores
Tipos de excepciones

Las excepciones son eventos anormales generados por el propio proceso (no por hardware externo). Se clasifican por qué tan recuperable es el error:

Fatal · irrecuperable
Abort
Error tan grave que el SO no puede saber exactamente qué instrucción lo causó ni puede reiniciar el programa. El proceso pasa directo a estado Saliente.
ej: corrupción interna · error de hardware crítico · valor ilegal en tablas del sistema
Recuperable · reintenta
Fault
Error que el SO puede corregir. Después de corregirlo, vuelve a ejecutar la misma instrucción que falló. El proceso no sabe que pasó nada.
Page fault: el proceso pidió leer una dirección de RAM que en realidad está en disco. El SO carga la página desde disco y reintenta la instrucción. El proceso continúa.
Continúa · siguiente instrucción
Trap
Se reporta después de la instrucción que la generó. El SO maneja el evento y el proceso continúa desde la instrucción siguiente (no la que falló).
Syscall open("archivo.txt"): el proceso ejecuta la instrucción de llamada, el SO abre el archivo, y el proceso continúa desde la instrucción siguiente con el resultado.
07 · Diseño del SO
¿Cómo ejecuta el propio SO?

El SO también es software — también necesita que el procesador lo ejecute. Existen tres enfoques para organizar esta relación, cada uno con analogías directas a los diseños de la Unidad 1:

I
Núcleo sin procesos
El SO es un bloque de código completamente separado de todos los procesos de usuario. Cada vez que un proceso es interrumpido, se hace un cambio de contexto completo al espacio del SO y luego de vuelta.
Analogía Unidad 1: diseño monolítico o en capas
II
Dentro del proceso de usuario
El SO se ejecuta en el contexto del proceso actual. Una syscall solo cambia el modo (usuario→kernel) dentro del mismo proceso, sin hacer un cambio de contexto completo. Más eficiente.
Analogía Unidad 1: diseño microkernel
III
SO basado en procesos
El SO es una colección de procesos de sistema independientes. Cada función principal (scheduler, driver de disco, etc.) es un proceso separado. Diseño modular con interfaces claras.
Analogía Unidad 1: diseño cliente-servidor
