CPU ociosa

El programador era el sistema operativo. Reservaba tiempo en una planilla, llegaba físicamente a la máquina, cargaba el compilador a mano, luego el programa en tarjetas perforadas, luego los datos. Ejecutaba, esperaba, recogía el resultado impreso. No había scheduler, no había abstracción. El procesador esperaba activamente al disco consultando en loop "¿terminaste?" — tiempo de CPU desperdiciado para siempre.

Ciclo interno
1
PC apunta a instrucción. FETCH: se carga en IR.
2
Instrucción de leer disco → el procesador envía orden y entra en busy-waiting: loop preguntando "¿terminaste?"
3

4

Problema 1
Planificación manual. Si el programa tarda más de lo reservado, los demás esperan parados.
Problema 2
Tiempo de setup enorme. Cargar compilador + programa + datos antes de cada corrida.
Por qué importa
El hardware costaba millones. Que la CPU esté ociosa el 97% del tiempo era inaceptable.

La gran revolución. Varios programas en memoria simultáneamente. Cuando A espera E/S, el SO cede la CPU a B. Cuando el disco termina, interrumpe al procesador — que en ese momento puede estar ejecutando B — para notificarle. El SO guarda el contexto de B y reanuda A exactamente donde lo había dejado. La CPU casi nunca queda ociosa.

Manejo interrupción
Ciclo con interrupciones
1
A ejecuta. Llega a syscall de disco → SO ordena al controlador y devuelve CPU a B sin esperar.
2
B corre. Al fin de cada instrucción, el procesador consulta el registro de interrupciones.
3
Disco termina → envía señal. El SO guarda contexto de B (PC, PSW, registros) en la process table.
4
SO restaura contexto de A exactamente como estaba. A continúa en la instrucción siguiente a su syscall.
5
DMA: el controlador transfiere datos a RAM sin molestar al procesador. Solo interrumpe al final.
La clave: context switch
El SO fotografia PC + todos los registros del proceso suspendido y los guarda en la process table. Restaurarlos es arrancar desde donde quedó.
Innovación: DMA
Direct Memory Access: el controlador de E/S transfiere datos directamente a RAM, sin pasar por el procesador. La CPU queda libre.
Resultado
Utilización de CPU sube a 60–90%. El mismo hardware procesa mucho más trabajo por hora.

respuesta percibida

Igual que multiprogramación, pero el temporizador de hardware interrumpe al procesador cada pocos milisegundos aunque el proceso actual no haya pedido E/S. Esto permite que múltiples usuarios interactivos compartan la CPU. La técnica funciona porque los humanos reaccionamos en cientos de milisegundos — la CPU puede atender a docenas de usuarios en ese tiempo.

Context switch (SO)
Ciclo con quantum de tiempo
1
Usuario 1 corre. Timer cuenta hacia atrás. Al expirar → interrupción de temporizador.
2
SO guarda contexto completo de U1 en process table (PC, PSW, registros, archivos abiertos).
3
Si U2 no entra en RAM libre → swap out: estado de U2 se escribe a disco. Solo si es necesario.
4
Se carga contexto de U2. PC apunta a donde U2 había quedado. Continúa. U2 no nota nada.
5
Con n=3, cada uno recibe 1/3 de CPU. A 2GHz y quantum de 20ms → 660M instrucciones por turno.
Por qué funciona
Los humanos reaccionamos en ~200ms. La CPU da vueltas completas entre todos los usuarios varias veces antes de que notemos algo.
Diferencia con multiprog.
Multiprogramación: cambio de proceso por E/S. Tiempo compartido: cambio de proceso por timer, aunque no haya E/S. Es proactivo.
Nuevos desafíos
Protección entre usuarios. Sistema de archivos con permisos. Conflictos de recursos (impresoras). El SO se complejiza significativamente.

Mecanismo fundamental
Los 4 tipos de interrupciones

Una interrupción suspende el ciclo normal de instrucciones para atender un evento. Son lo que hace posible que el SO retome el control de la CPU.

⚡

De programa

Generadas por la ejecución de una instrucción del propio programa.

ej: división por cero · overflow aritmético · syscall · acceso a memoria inválida
💾

De E/S

Generada por el controlador de un dispositivo para indicar que terminó una operación.

ej: disco terminó de leer · impresora lista · teclado tiene dato
⏱

De temporizador

El reloj del sistema interrumpe cada N milisegundos. Esencial para el tiempo compartido.

ej: quantum expirado → forzar context switch aunque no haya E/S
🔴

Fallo de hardware

Error físico irrecuperable que requiere atención inmediata del SO.

ej: error de paridad en RAM · fallo de fuente · temperatura crítica
